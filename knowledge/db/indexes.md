# Индексы в реляционных базах данных

## 1. Введение

В реляционных базах данных индекс представляет собой специализированную структуру данных, предназначенную для ускорения операций поиска и доступа к строкам таблицы.

Без использования индексов СУБД выполняет последовательный просмотр таблицы (Full Table Scan), что приводит к линейной сложности O(n). При увеличении объема данных это становится критическим узким местом.

Индекс позволяет сократить сложность поиска до логарифмической или даже константной, в зависимости от его типа.

---

## 2. Принцип работы индекса

Индекс хранит:

* значение индексируемого столбца
* ссылку (указатель) на строку таблицы

Важно понимать, что индекс:

* является отдельной структурой
* требует дополнительной памяти
* должен поддерживаться в актуальном состоянии при изменении данных

Каждая операция вставки, обновления или удаления данных приводит к обновлению всех связанных индексов.

---

## 3. Когда использование индексов оправдано

Индексы эффективны в следующих случаях:

* использование столбца в условиях WHERE
* участие столбца в JOIN
* сортировка через ORDER BY
* агрегация и группировка (GROUP BY)
* высокая селективность данных (много уникальных значений)

Индексы неэффективны:

* для маленьких таблиц
* для столбцов с низкой уникальностью (например, boolean)
* при преобладании операций записи над чтением

---

## 4. B-Tree индекс

### 4.1. Общая характеристика

B-Tree (сбалансированное дерево) является стандартным типом индекса в большинстве СУБД, включая PostgreSQL, MySQL и Oracle Database.

Это универсальная структура, обеспечивающая эффективный поиск, вставку и удаление.

---

### 4.2. Структура

B-Tree представляет собой дерево, в котором:

* каждый узел содержит отсортированные ключи
* дерево остается сбалансированным
* все листья находятся на одном уровне

Поиск выполняется сверху вниз, от корня к листу.

---

### 4.3. Сложность операций

* поиск: O(log n)
* вставка: O(log n)
* удаление: O(log n)

---

### 4.4. Поддерживаемые операции

B-Tree индекс поддерживает:

* точечный поиск (=)
* диапазоны (<, >, BETWEEN)
* сортировку (ORDER BY)
* префиксный поиск (LIKE 'abc%')

---

### 4.5. Пример

```sql
CREATE INDEX idx_users_age ON users(age);
```

---

### 4.6. Особенности

* хранит данные в отсортированном виде
* эффективно работает с диапазонами
* используется по умолчанию

---

### 4.7. Ограничения

* не оптимален для сложных структур (JSON, массивы)
* менее эффективен, чем hash, для строго точечного поиска

---

## 5. Hash индекс

### 5.1. Общая характеристика

Hash индекс основан на применении хеш-функции к значению ключа.

---

### 5.2. Принцип работы

Алгоритм:

1. значение преобразуется в hash
2. hash используется как адрес
3. выполняется прямой доступ к данным

---

### 5.3. Сложность

* поиск: O(1) в среднем

---

### 5.4. Поддерживаемые операции

Только:

* точное сравнение (=)

---

### 5.5. Ограничения

Hash индекс не поддерживает:

* диапазонные запросы
* сортировку
* операции LIKE
* ORDER BY

Причина – потеря порядка данных после хеширования.

---

### 5.6. Пример

```sql
CREATE INDEX idx_users_email ON users USING HASH(email);
```

---

### 5.7. Практическое использование

В реальных системах применяется редко, поскольку:

* B-Tree покрывает большинство сценариев
* преимущества hash незначительны
* в некоторых СУБД есть ограничения по надежности

---

## 6. GIN индекс

### 6.1. Общая характеристика

GIN (Generalized Inverted Index) предназначен для индексирования составных и сложных данных.

Наиболее активно используется в PostgreSQL.

---

### 6.2. Принцип работы

GIN хранит отображение:

ключ → список строк

Например:

```
"apple" → [1, 7, 15]
"banana" → [2, 7]
```

Это называется инвертированным индексом.

---

### 6.3. Области применения

GIN применяется для:

* полнотекстового поиска
* JSON / JSONB
* массивов
* поиска по множествам

---

### 6.4. Пример

```sql
CREATE INDEX idx_users_bio 
ON users USING GIN(to_tsvector('english', bio));
```

---

### 6.5. Преимущества

* высокая эффективность для сложных данных
* поддержка поиска по множественным значениям

---

### 6.6. Недостатки

* большой объем памяти
* медленные операции вставки
* сложность поддержки

---

## 7. Составные индексы (Composite Index)

### 7.1. Определение

Составной индекс строится по нескольким столбцам:

```sql
CREATE INDEX idx_users_age_salary 
ON users(age, salary);
```

---

### 7.2. Правило левого префикса

Индекс работает только если используется его начальная часть:

Работает:

```sql
WHERE age = 30
WHERE age = 30 AND salary > 5000
```

Не работает:

```sql
WHERE salary > 5000
```

---

### 7.3. Практическое значение

Порядок полей в индексе критически важен и должен соответствовать реальным запросам.

---

## 8. Покрывающий индекс (Covering Index)

Индекс называется покрывающим, если он содержит все поля, необходимые для выполнения запроса.

Пример:

```sql
CREATE INDEX idx_users_age_name 
ON users(age, name);
```

Запрос:

```sql
SELECT name FROM users WHERE age = 30;
```

В этом случае СУБД не обращается к таблице, а работает только с индексом.

---

## 9. Стоимость индексов

### 9.1. Память

Каждый индекс занимает дополнительное место на диске.

---

### 9.2. Запись

При операциях:

* INSERT
* UPDATE
* DELETE

индексы должны обновляться.

Это приводит к:

* увеличению времени записи
* росту нагрузки

---

### 9.3. Баланс

Индекс – это компромисс:

* ускорение чтения
* замедление записи

---

## 10. Анализ выполнения запросов

Для понимания использования индексов применяется команда:

```sql
EXPLAIN ANALYZE
```

Она показывает:

* используется ли индекс
* стоимость запроса
* план выполнения

---

## 11. Типичные ошибки

1. Индексация всех столбцов
2. Игнорирование EXPLAIN
3. Неправильный порядок полей в составных индексах
4. Индексация низкоселективных полей
5. Отсутствие анализа реальных запросов

---

## 12. Сравнение типов индексов

| Характеристика  | B-Tree  | Hash   | GIN     |
| --------------- | ------- | ------ | ------- |
| Поиск =         | да      | да     | да      |
| Диапазоны       | да      | нет    | нет     |
| Сортировка      | да      | нет    | нет     |
| JSON            | нет     | нет    | да      |
| Full-text       | нет     | нет    | да      |
| Универсальность | высокая | низкая | средняя |

---

## 13. Практические рекомендации

1. Использовать B-Tree по умолчанию
2. Добавлять индекс только под конкретный запрос
3. Проверять через EXPLAIN
4. Избегать избыточных индексов
5. Учитывать нагрузку на запись
6. Использовать GIN для JSON и текстового поиска
