# Полиморфизм, типы ссылок и абстракции в Java

## Общая характеристика

В языке Java объектная модель основана на разделении:

* **типа ссылки (reference type)**;
* **фактического типа объекта (runtime type)**.

Это разделение является основой:

* полиморфизма;
* работы с абстракциями (интерфейсами и базовыми классами);
* гибкости архитектуры.

---

## Тип ссылки и фактический тип объекта

Рассмотрим выражение:

```java
A obj = new B();
```

В данном случае:

* `A` – тип ссылки (определяется на этапе компиляции);
* `B` – фактический тип объекта (определяется во время выполнения).

---

### Семантика

Тип ссылки определяет:

* какие методы и поля доступны на этапе компиляции;

Фактический тип объекта определяет:

* какая реализация метода будет вызвана во время выполнения.

---

### Иллюстрация

```text
obj ─────► [ объект B ]
            ↑
         содержит A + B
```

---

## Доступ к членам объекта

### Методы

```java
obj.method();
```

* доступность метода определяется типом ссылки (`A`);
* реализация метода выбирается по фактическому типу (`B`).

Это называется **динамическое связывание (dynamic binding)**.

---

### Поля

```java
obj.field;
```

* доступ определяется типом ссылки (`A`);
* значение берётся из класса `A`.

Поля:

* не участвуют в полиморфизме;
* не переопределяются, а **скрываются (field hiding)**.

---

### Вывод

| Элемент | Поведение                  |
| ------- | -------------------------- |
| Методы  | динамическое (runtime)     |
| Поля    | статическое (compile-time) |

---

## Полиморфизм

### Определение

Полиморфизм – способность использовать объекты разных классов через общий тип с возможностью различного поведения.

---

### Пример

```java
A obj = new B();
obj.print();
```

Вызывается реализация `B.print()`.

---

### Ключевое свойство

Полиморфизм в Java применяется:

* к методам;
* не применяется к полям.

---

## Ограничение через тип ссылки

Тип ссылки ограничивает доступ:

```java
A obj = new B();
```

Допустимо:

```java
obj.methodFromA();
```

Недопустимо:

```java
obj.methodFromB(); // ошибка компиляции
```

---

### Приведение типов (casting)

```java
((B) obj).methodFromB();
```

Допустимо, если объект действительно является `B`.

---

### Риск

```java
A obj = new A();
((B) obj); // ClassCastException
```

---

## Абстракция: контракт и реализация

### Контракт

Контракт определяет **что можно делать**:

```java
interface NotificationService {
    void send(String message);
}
```

---

### Реализация

Реализация определяет **как это делается**:

```java
class EmailService implements NotificationService {

    void connectToServer() {}
    void formatMessage() {}

    @Override
    public void send(String message) {
        connectToServer();
        formatMessage();
    }
}
```

---

### Ключевое различие

```text
Интерфейс = что делать
Класс = как делать
```

---

## Инкапсуляция реализации

Методы, являющиеся деталями реализации:

* не должны попадать в интерфейс;
* используются только внутри класса.

---

### Неправильный интерфейс

```java
interface NotificationService {
    void send();
    void connectToServer();
}
```

Недостатки:

* нарушает абстракцию;
* делает контракт избыточным;
* снижает гибкость.

---

### Правильный интерфейс

```java
interface NotificationService {
    void send();
}
```

---

## Причины использования абстракций

### Снижение связности

Код зависит от контракта, а не от реализации.

---

### Заменяемость реализаций

```java
NotificationService service = new EmailService();
service = new SmsService();
```

---

### Расширяемость

Добавление новых реализаций не требует изменения существующего кода.

---

### Тестируемость

Можно подставлять mock-реализации.

---

## Классы и интерфейсы

### Общность

И классы, и интерфейсы могут использоваться как тип ссылки:

```java
A obj = new B();
```

---

### Отличия

| Характеристика | Интерфейс            | Класс        |
| -------------- | -------------------- | ------------ |
| Назначение     | контракт             | реализация   |
| Состояние      | отсутствует (обычно) | присутствует |
| Наследование   | множественное        | одиночное    |
| Поведение      | декларация           | реализация   |

---

### Практическое различие

Классы используются, когда:

* требуется общее поведение;
* есть состояние;

Интерфейсы используются, когда:

* требуется абстракция;
* есть несколько реализаций.

---

## Практический пример

```java
interface PaymentService {
    void pay(int amount);
}
```

```java
class CardPayment implements PaymentService {
    public void pay(int amount) {}
}

class CryptoPayment implements PaymentService {
    public void pay(int amount) {}
}
```

Использование:

```java
PaymentService service = new CardPayment();
```

---

## 11. Основные ошибки

1. Добавление в интерфейс методов, относящихся к реализации
2. Использование конкретных классов вместо абстракций
3. Непонимание различия между полями и методами
4. Злоупотребление приведением типов
